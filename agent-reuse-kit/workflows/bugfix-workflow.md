# 通用 Bug 修复工作流

这个工作流适合复制到新项目的 `.agents/skills/bugfix/SKILL.md`，再按项目实际情况替换路径、命令和技术栈。

## 0. 启动规则

- 只有当用户要求修复、诊断、复现、回归验证或修复后复盘时使用。
- 如果用户只问原因或要代码审查，先回答问题或按 review 方式给发现；除非用户明确要修，否则不要擅自改代码。
- 如果用户说“继续”“直接修”“帮我处理”，默认执行诊断、修复和验证，不停在方案阶段。
- 缺少复现信息时，先从代码、测试、日志和文档推断；只有关键事实无法推断且继续有风险时，再问一个简短问题。

## 1. 项目边界

- 遵守仓库根目录 `AGENTS.md`。
- 不自动启动 dev server。
- 不自动运行完整构建。
- 不直接修改构建产物目录。
- 不自动创建 Git 提交、分支、PR 或 Issue。
- 不回滚用户已有改动。
- 新文件、测试文件、临时脚本放到项目约定目录。

## 2. 快速定位

- 搜索文件：`rg --files`
- 搜索内容：`rg "关键词" <路径>`
- 读取文件：使用当前工具的只读命令。
- 多个只读命令可以并行执行。
- 修改文件使用局部 diff。

建议先查：

- 报错栈里的文件。
- 相关测试。
- 相关功能文档。
- 最近修改过或用户提到的文件。
- 同类功能的既有实现。

## 3. 诊断顺序

1. 入口层：路由、页面入口、命令入口、配置入口是否正确。
2. 状态层：共享状态、缓存、响应式引用、生命周期是否被错误使用。
3. 数据层：数据结构是否同步，边界输入是否处理。
4. 交互层：props、emit、事件、表单、拖拽、异步竞态是否正确。
5. 规则与算法层：优先级、过滤条件、随机性、冲突处理是否正确。
6. UI/样式层：布局、滚动、尺寸、颜色、图标、移动端约束是否正确。
7. 平台层：Web、桌面端、服务端、部署平台差异是否被隔离。

## 4. 修复方式

- 先理解现有模式，再做最小改动。
- 优先修源文件，不改生成产物。
- 修改现有文件前先读取相关片段。
- 遇到用户已有改动时，保留并绕开；同一文件内相关时读清楚再合并。
- 不为了一个 Bug 引入大范围重构。
- 新抽象必须有实际收益。
- UI 修复同时检查文字溢出、控件尺寸稳定、颜色和图标规范。

## 5. 测试验证

优先运行和改动相关的最小测试，例如：

```bash
<单个测试命令>
<相关测试套件命令>
```

必要时运行：

```bash
<完整测试命令>
<类型检查命令>
```

注意：

- 不自动运行构建命令。
- UI 修复如果已有 dev server，可以用浏览器验证。
- 如果没有 dev server，说明需要用户手动启动后再做浏览器验证。
- 修复共享逻辑、算法、数据结构时，优先补充或更新测试。

## 6. 文档与沉淀

- 只有修复改变了业务规则、数据模型、架构约定或易错点时，才更新文档。
- 优先更新 `.agents/features/` 中对应功能文档。
- 用户可见功能、入口、工作流、限制条件或排查步骤变化时，同步更新用户手册或帮助文档。
- 不创建临时总结文档作为长期维护资料。

## 7. 收尾格式

最终回复简洁说明：

- 修了什么问题，涉及哪些关键文件。
- 运行了哪些测试，结果如何。
- 哪些验证没有做，以及原因。
- 如需用户手动验证，给出具体步骤。

不要声称已构建、已启动服务、已提交或已创建 Issue，除非实际完成了对应操作。
