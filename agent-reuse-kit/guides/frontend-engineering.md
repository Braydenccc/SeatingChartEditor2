# 前端工程与 UI 经验

这份文档提取当前项目中可迁移的前端协作经验。它不是某个框架的完整规范，而是给 agent 和开发者共同遵守的默认偏好。

## 技术选择

- 沿用项目已有框架和状态管理方式，不因为新功能就引入全新架构。
- 新状态优先放到对应功能模块或既有状态容器中。
- 跨平台项目要隔离平台能力：业务组件不要直接调用 Node.js、桌面端 SDK 或部署平台私有 API。
- 结构化数据优先使用解析器、类型或 schema，不用临时字符串拼接。

## 文件与目录

- 新文件优先使用项目当前主语言和类型系统。
- 类型定义集中放到约定目录。
- 临时测试脚本放到临时目录，不放进正式源码目录。
- 不直接修改构建产物目录。
- 功能文档放 `.agents/features/`，长期用户文档放项目约定的用户手册位置。

## UI 规范

- 颜色使用语义化设计变量，例如 `var(--color-*)`，不要在组件中硬编码十六进制、RGB 或 RGBA。
- 新增颜色变量要同时考虑浅色和深色主题，并使用语义命名。
- 图标使用统一图标库，按需具名导入，通过组件 prop 控制尺寸。
- 不用内联 SVG 或 Unicode 字符充当图标按钮。
- 图标按钮设置固定尺寸、`padding: 0`、`display: flex`、居中对齐。
- 控件高度、网格、棋盘、工具栏等固定格式元素要有稳定尺寸，避免内容变化导致布局跳动。
- 文案要在移动端和桌面端都不溢出、不遮挡、不覆盖邻近元素。
- 页面应优先呈现实际可用的产品界面，而不是只有说明和装饰。

## 交互与状态

- 对可能清空、覆盖或批量修改用户数据的操作弹出确认。
- 拖拽、弹窗、异步请求和编辑会话需要在失败、取消或组件卸载时清理状态。
- 点击项内部如果有按钮、输入框或菜单，要避免点击穿透触发父级选择逻辑。
- 配置变更要明确处理旧数据、旧缓存和旧选择状态。

## 文档同步

- 用户可见功能、入口、工作流、限制条件或排查步骤变化时，同步更新用户手册或帮助文档。
- 代码里的功能说明不要和用户手册形成两份真源。
- 复杂模块新增或重构后，补充 `.agents/features/`：职责、入口、数据流、常见陷阱、测试入口。
- 不把临时修复总结沉淀成长期文档。

## 验证策略

- 先运行和改动相关的最小测试。
- 共享逻辑、算法和数据结构改动需要测试覆盖。
- UI 改动优先在已有 dev server 上做浏览器检查。
- 没有 dev server 时，不擅自启动；在收尾中说明需要用户启动后验证。
- 完整构建属于高成本验证，除非用户要求或任务明确需要，否则不自动运行。
