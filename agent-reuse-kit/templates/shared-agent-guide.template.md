## 代码规范

- 修改前先读相关文件和邻近实现。
- 优先沿用既有模式，不为小改动引入新架构。
- 保持改动范围小，避免顺手重构无关代码。
- 新抽象必须能减少真实重复、降低复杂度或匹配已有模式。
- 结构化数据优先使用结构化 API 或解析器，不用脆弱字符串处理。
- 命名、目录、语言后缀按项目当前约定执行。

## UI 与前端规范

- 颜色使用项目设计变量，不在组件中硬编码颜色。
- 图标使用项目统一图标库，按需导入，不用内联 SVG 或 Unicode 字符冒充图标。
- 控件尺寸要稳定，避免 hover、加载文案、图标或动态内容导致布局跳动。
- 移动端和桌面端都要检查文字溢出、遮挡和滚动区域。
- 不在应用内写“如何使用本功能”的说明文字，除非该界面本身就是帮助或文档。
- 用户可见流程变化要同步更新用户文档或应用内手册。

## 测试与验证

优先级：

1. 运行最小相关测试。
2. 共享逻辑、算法、数据结构变更时补充或更新单元测试。
3. UI 变更在已有 dev server 时用浏览器验证；没有 dev server 时说明需要用户启动。
4. 高风险改动再运行更大范围测试或类型检查。

收尾时明确说明：

- 已运行哪些验证。
- 哪些验证没做以及原因。
- 需要用户手动检查的具体步骤。

## 文档沉淀

- `.agents/features/` 用来记录稳定模块知识，不记录临时过程。
- 功能文档建议包含：职责、入口文件、数据流、常见陷阱、测试入口。
- 不创建临时总结文档作为长期维护资料。
- 如果文档由脚本生成，只修改真源，再运行同步命令。
